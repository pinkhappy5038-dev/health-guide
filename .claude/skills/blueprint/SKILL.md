# blueprint — 기획 인터뷰 스킬

> 목적: 사용자(비개발자·입문 바이브코더)의 아이디어를 **아주 자세한 질문**으로 파고들어,
> 개발을 바로 시작할 수 있는 설계 문서 7종 + 화면별 명세 폴더(screens/)를 만든다. **질문 없이 산출물부터 쓰는 것 금지.**
>
> **설계 승인 게이트**: 이 인터뷰가 끝나 산출물이 승인되기 전에는 코드를 쓰지 않는다. 아무리 간단해 보여도.
> 이 게이트는 부탁이 아니다 — plan-gate 훅이 plan.md 가 채워지기 전의 apps/ 코드 작성을 물리적으로 차단한다.

## 절차 (반드시 이 순서)

> **부분 발동 (전체 8단계의 예외)**: 기획이 이미 완성돼 있는데(docs/plan/ 존재) 사용자가 특정 주제만 원하면
> (예: "DB 설계 도와줘"·"화면만 다시 보자"·"연동 하나 추가하자") 8단계를 다시 돌지 않는다.
> ① 기존 산출물을 선조사(규칙 10)하고 ② 해당 단계만 카드 인터뷰로 손본 뒤 ③ 그 단계 산출물(예: 05-db.md)과
> plan.md·06-decisions.md(바뀐 결정)를 갱신하고 ④ 허점 라운드는 바뀐 부분만 최대 3문으로 짧게 돈다.
> 기획이 아예 없으면(docs/plan/ 없음) 부분 요청이어도 "전체가 먼저"라고 말하고 8단계로 간다.
> 빌드가 시작된 뒤의 기능 추가·변경이면 킷 CLAUDE.md 의 **변경 제안서 규칙**(docs/plan/changes/)과 짝으로 돈다 — 제안서 승인 후, 설계에 닿는 변경(화면·DB·연동)은 이 부분 발동으로 해당 산출물을 갱신한다.

### 0. 시작
- 사용자가 아이디어를 줬으면 그것을 1단계 답의 초안으로 삼는다. 없으면 1단계부터 묻는다.
- **profile.md 가 채워져 있으면 그것부터 분해하라** — "만들고 싶은 것"→①·②단계, 프로젝트 조건(최종 형태·결제·실시간·저장·로그인·화면 수)→③·⑤·⑦·⑧단계 초안으로 매핑하고, 해당 질문은 백지 대신 "신청폼에 이렇게 쓰셨던데, 지금도 맞습니까?" 확인 카드로 묻는다. 스코프 판정이 "조정하면 가능/크게 줄여야"면 ③단계 스코프컷에서 그 이유를 함께 다룬다.
- **기획 자료를 길게 줬으면(문서·메모·대화 붙여넣기) 바로 묻지 말고 먼저 분해하라:**
  - **요소 파악** — 자료에서 타겟→②단계, 기능→③, 흐름→④, 화면→⑤, 데이터→⑥, 연동→⑦, 권한/운영→⑧ 로 추출해 각 단계 답 초안에 매핑한다.
  - **단어 파악** — 자료에 나온 업계 용어·서비스명·애매한 표현("자동으로", "관리")은 뜻을 짐작하지 말고 해당 단계에서 확인 질문으로 푼다.
  - **문맥 파악** — 자료끼리 모순되거나 빠진 결정(누가 보나·지우면·실패하면)은 허점 후보로 적어뒀다가 해당 단계와 허점 라운드에서 따진다.
  - 채워진 단계는 "이렇게 이해했는데 맞습니까?" 확인 카드로 빠르게 통과하고, 빈 단계만 백지 질문한다. **자료를 줬다고 단계를 건너뛰지 마라** — 확인이 질문보다 빠를 뿐, 8단계는 전부 거친다.
- 선언하라: "8단계 질문으로 설계도를 만든다. 한 단계씩 간다. 대답이 곧 문서가 된다." **그리고 8단계가 뭔지 지도를 한 줄씩 보여줘라** (①아이디어 한 줄 ②누가 쓰나·무슨 문제 ③핵심 기능 고르기 ④사용자가 거치는 길 ⑤화면 밑그림 ⑥뭘 저장하나 ⑦밖에서 뭘 빌리나 ⑧권한·운영 규칙). "8단계 설계 틀" 같은 말만 던지고 내용을 안 보여주면 사용자는 지도 없이 끌려간다.
- 이후 매 단계를 시작할 때 "지금 N/8단계 — [단계 이름]"을 먼저 말해 어디까지 왔는지 보여준다.
- 선언한 그 턴에서 바로 1단계 첫 질문을 AskUserQuestion 카드로 던진다. 설명만 하고 카드 없이 턴을 끝내지 마라.

### 1~8. 인터뷰 (질문 은행 = `references/question-bank.md`)
단계: ① 아이디어 한 줄 ② 타겟·문제 ③ 핵심기능·스코프컷(MVP 3개 · **감당선 판정**(기능마다 감당 가능/조정/감당선 밖 — 넷플릭스는 못 만든다를 근거로) · **완성 제품 로드맵**(MVP→다음 버전→완성 제품, 꿈은 층으로 보존)) ④ 유저플로우 ⑤ 화면·와이어프레임 ⑥ 데이터 모델(DB) ⑦ 외부 연동·키 — **묻기 전에 먼저 그려라**: 지금까지 답을 근거로 "시스템 지도 초안"(벽 안 = 홀(화면)·주방(서버)·창고(DB) / 벽 밖 = 필요해 보이는 문 후보들: 번역·지도·결제·로그인 등)을 글 도식으로 보여주고, 문 하나씩 카드로 확정한다 — 비개발자는 이 지도를 보고서야 자기 앱에 뭐가 필요한지 감을 잡는다. **Supabase·Vercel 은 기본 스택(안 묻고 자동 포함), Cloudflare 는 도메인(DNS)·이미지 저장량(R2) 문답으로 조건부 포함** ⑧ 지킬 조건(권한·보안·실패 대비·운영 — 개발자 말로 '비기능'. 앱이 '하는 일'이 아니라 '지켜야 할 규칙': 식당의 메뉴가 아니라 영업 규칙·위생·소방)

**조립 블럭 4종 — 이 인터뷰의 최종 목표는 "블럭 목록 확정"이다:**
앱은 레고처럼 네 종류 블럭으로 조립된다. 인터뷰는 이 부품 목록을 채우는 과정이다.
| 블럭 | 뭔가 | 어느 단계에서 확정 |
|---|---|---|
| **공간** | 화면 덩어리 — 손님용 웹 · 관리자 페이지 · 파트너용 화면 | ②의 역할에서 나와 ⑤ 맨 앞에서 확정 |
| **역할** | 사람 — 누가 쓰고, 누가 안의 내용을 넣고 고치나 | ② |
| **창고** | 저장하는 데이터 표들 | ⑥ |
| **문** | 빌려오는 외부 서비스 (번역·지도·결제·로그인 등) | ⑦ |

**매 단계 요약 때 "조립 블럭 현황"을 누적해 보여줘라** (예: "지금까지 모인 블럭 — 공간 2(손님 웹·관리자) · 역할 3 · 창고 표 4 · 문 5"). 부품이 쌓이는 게 눈에 보여야 사용자가 자기 앱의 구조를 손에 잡는다. 특히 "관리자 페이지가 필요하다"는 ②~⑤에서 반드시 수면 위로 올린다 — 빌드 중에 발견하면 일이 두 배가 된다.

**인터뷰 규칙 (강제):**
1. **한 번에 한 단계.** 단계 안에서도 질문을 2~4개씩 끊어 묻는다 (한 번에 16개 던지지 않는다).
2. **필수 질문에 빈 답이 있으면 다음 단계로 못 넘어간다.** "모르겠다"도 답이다 — 단, 그 경우 질문 은행의 예시 답을 보여주고 한 번 더 묻는다. 그래도 모르면 "미정"으로 기록하고 문서에 `[미정]` 표시.
   - **카드가 취소·미응답으로 접혀도 마찬가지다**: 답을 받은 것처럼 진행하는 것 금지. ① 지금까지 확정된 것과 남은 카드를 3줄로 요약하고 ② "돌아와서 '이어서'라고 하면 그 카드부터 다시 띄운다"고 안내하고 멈춰라. 사용자가 딴 주제를 꺼내면 그걸 먼저 처리하되, 끝나면 남은 카드로 복귀를 제안한다.
3. **조건부 질문**은 앞 답이 조건에 해당할 때만 묻는다 (은행에 조건 명시돼 있음).
4. **단계가 끝나면 그 단계 산출물을 3~6줄로 요약해 보여주고 "맞습니까?" 확인** 후 다음 단계로.
5. **질문은 AskUserQuestion 도구로 제시한다(선택 카드 UI).** 두 가지가 핵심이다:
   - **선택지는 미리 정해두지 않고 추론 생성한다.** 사용자의 아이디어·업종·앞 단계 답변 맥락에서 그럴듯한 선택지 2~4개를 그때그때 만든다. 질문 은행의 예시 선택지는 참고일 뿐, 그대로 박지 마라. (예: 카드뉴스 앱이면 "누구를 위해?"에 [동네 자영업자 / 인플루언서 / 직장인 부업 / 비영리 단체] 처럼 그 맥락에 맞게 생성.)
   - **기타(자유 입력)는 항상 열려 있다.** AskUserQuestion 이 자동으로 "기타" 를 제공하니, 선택지에 없는 답을 사용자가 직접 적을 수 있다. 그래서 서술형에 가까운 질문(타겟·문제 등)도 선택지 몇 개 + 기타 형태로 물을 수 있다.
   - **고르는 질문은 예외 없이 AskUserQuestion으로 묻는다. 텍스트로 질문만 적고 턴을 끝내는 것 금지.** 단 하나의 예외 — **나열형 브레인스토밍**(3-1 기능 쏟아내기, 6단계 명사 전수추출처럼 많이 나올수록 좋은 질문)은 텍스트로 받아도 된다: 그 턴에 그 질문 하나만 던지고, 답을 받으면 즉시 정리·판정 카드로 이어간다. (텍스트 질문은 사용자가 안 답하고 지나칠 수 있지만, 카드는 답하기 전엔 진행이 안 된다.) 1-1 아이디어 한 줄 같은 개방형도 카드로 묻는다: 선택지에 대표 유형 3~4개(예: 예약·신청 받기 / 기록·관리 도구 / 홍보 페이지 / 판매·결제)를 놓고, 질문 본문에 "떠오르는 문장이 있으면 '기타'에 그대로 적어주세요"를 반드시 넣는다. 여러 항목을 한 화면에 물을 땐 AskUserQuestion 한 번에 질문 2~4개를 묶어도 된다.
   - **특히 ⑥ 데이터 모델은 전 질문을 AskUserQuestion 으로만 진행한다(텍스트 서술 금지) — 은행의 "6단계 카드 흐름"을 따른다.**
6. 전문용어가 나오면 그 자리에서 한 줄 비유로 풀어라 (예: RLS = 창고 문 앞의 출입 규칙).
7. 사용자가 "빨리 넘어가자"고 해도 ⑥ 데이터 모델의 필수 질문은 생략 불가 — "여기서 아낀 10분이 나중에 10시간이 된다"고 말하고 진행한다.
8. **펜을 돌려줘라.** 너는 사용자 대신 결정하지 않는다. "X를 Y로 이해했는데 맞습니까?"는 되고, 사용자 대신 기능을 발명하거나 MVP를 네가 잘라주는 것은 안 된다. 네가 저작하기 시작했다고 느끼면 멈추고 질문으로 돌아가라. (학습 목적: AI가 다 정해주면 사용자가 자기 프로젝트를 이해 못 한다.)
9. **추천 답을 1순위로.** 앞 단계 답·업종 맥락에서 답을 추정할 수 있으면 첫 선택지에 "(추천)"을 붙이고 근거를 한 줄 단다. 사용자는 "맞아"만 눌러도 굴러가야 한다 — 모르는 사람일수록 백지 질문이 무섭다. (단 추천은 제안이지 결정이 아니다 — 규칙 8과 충돌하면 8이 이긴다.)
10. **묻기 전에 먼저 찾아라.** 기존 profile.md(신청폼)·plan.md·docs/plan/*·코드·.env.example 에 이미 답이 있으면 백지 질문 대신 "지금 X로 돼 있던데, 유지합니까?" 확인 질문으로 바꾼다. 이미 아는 걸 물으면 사용자는 인터뷰를 신뢰하지 않는다.

### 9. 산출물 생성
- 8단계가 끝나면 `references/templates.md` 의 형식대로 7개 파일을 `docs/plan/` 에 쓴다. `07-architecture.md` 의 도식엔 ⑦에서 확정된 문만 그린다(후보였다가 뺀 건 "안 여는 문"에).
- `docs/plan/screens/` 에 **화면마다 파일 하나**를 쓴다 (⑤ 답 + 5-7 예외 스윕 결과로 — 02-spec 표와 어긋나면 안 된다). 03-userflow.md 끝에 화면 색인 표를 넣고, plan.md 의 화면 스텝 줄 끝에 `(명세: docs/plan/screens/S1-….md)` 참조를 단다.
- 루트 `plan.md` 를 답변 내용으로 갱신한다. **조립 순서는 2층**: 블럭 = MVP 기능 하나(3-6 순서, `### 블럭 N. 기능명 (완료 기준)` 제목) / 스텝 = 그 안의 체크박스(한 번에 시킬 크기, 줄 끝에 `— 확인: 눈으로 보는 방법`). **블럭 3원칙**: ① 끝나면 눈에 보이는 결과 하나 ② 앞 블럭만 있으면 되고 뒤 블럭을 기다리지 않음 ③ 이 블럭에 필요한 것만(전체 DB를 첫 블럭에 만들지 않기). 스텝은 블럭당 3~5개.
- `docs/plan/06-decisions.md` 를 초기화한다: 인터뷰에서 나온 갈림길 결정(스코프컷으로 뺀 것, 연동 선택, 권한/삭제 정책, 추천을 뒤집은 것)을 결정당 4줄(결정·왜·버린 대안·영향)로 기록 — 최소 3건, 상한 없음.
- ⑦ 답에 나온 연동의 키가 `.env.example` 에 없으면 추가한다 (**`.env` 는 절대 건드리지 않는다**). 연동 목록엔 기본 스택(Supabase·Vercel)이 항상 들어가고, Cloudflare 는 7-7/7-8 판정에 해당할 때만 사유와 함께 들어간다. 외부 서비스는 전부 실명으로 적는다.
- 모순 검사: 화면에 있는데 저장 안 하는 데이터 / 저장하는데 아무도 못 보는 데이터 / 버튼의 목적지 없는 화면 — 발견하면 해당 단계로 돌아가 되묻는다.

### 9.5 허점 라운드 (따져보기)
산출물을 쓴 직후, 빌드로 넘어가기 전에 **검증 인터뷰 한 라운드**를 돈다. 만들 때의 질문이 아니라 완성본을 두들기는 질문이다.
- 문서 전체에서 허점 후보를 심각한 순으로 뽑는다: **빈 결정**(실패하면? 지우면 어떻게 되나? 누가 보나?) · **문서 간 모순**(위 모순 검사에서 걸린 것 포함) · **뜬구름 표현**("잘", "적절히", "나중에"). 특히 05-db.md 의 관계·권한·삭제정책에 구멍이 없는지 본다.
- **한 번에 하나씩**, AskUserQuestion 카드로 따져 묻는다 (추천 답 1순위 — 규칙 9). 답이 나올 때마다 해당 문서에 즉시 반영한다.
- **최대 5문.** 사용자가 "그만"이라 하면 즉시 멈춘다. 끝나면 바뀐 것과 남은 구멍(`[미정]`)을 3줄로 요약한다. 라운드에서 결정이 바뀌었으면 `06-decisions.md` 에도 4줄로 쌓는다.
- 말투: 몰아세우지 않는다. 개업 전날 선배가 가게를 한 바퀴 돌며 "이건 어쩔 건데?" 묻는 톤. 여기서 아낀 10분이 나중에 10시간이다.
