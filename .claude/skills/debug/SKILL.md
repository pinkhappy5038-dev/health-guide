# debug: 버그 잡자

한 줄 정의: 증상에서 원인을 역추적해 딱 한 군데만 고치고 다시 확인하는 절차.
생활 비유: 물이 샐 때 벽을 다 뜯지 않는다. 어디서 새는지 먼저 찾고, 그 이음새 하나만 조인다.

## 철칙 (Iron Law)

**원인을 찾기 전엔 고치지 않는다.** 짐작으로 여기저기 손대면 새 버그가 생긴다. "아마 여기겠지"는 원인이 아니다.

## 절차

### 1. 증상 3줄 받기
사용자에게 이렇게 말하며 묻는다: "고치기 전에 상황부터 정확히 볼게요. 세 가지만 알려주세요."
- 무엇을 했나 (누른 버튼·입력한 값)
- 기대한 결과
- 실제 결과 (에러 글씨는 요약 말고 원문 그대로 붙여주세요)

### 2. 재현 확인
말하며 진행: "정말 매번 그러는지 한 번 더 해볼게요. 어쩌다 한 번이면 원인이 다르거든요."
- 같은 순서로 다시 해서 매번인지, 가끔인지 확인한다. 안 되는 화면을 직접 열어 눈으로 본다.
- 자동 테스트가 프로젝트에 있으면 **지금(고치기 전) 한 번 돌려 빨간 것을 확보**한다. 수정 후 같은 테스트가 초록이 되는 것이 가장 강한 증거다(전/후 비교).

### 3. 원인 후보 1개 찾기
말하며 진행: "증상에서 거꾸로 짚어볼게요. 에러 문구·안 뜨는 자리를 실마리로 관련 파일을 읽습니다."
- 증상과 직접 이어지는 코드만 읽고, 가장 그럴듯한 원인 후보 하나를 특정한다. (아직 안 고친다.)

### 4. 경로 점검 (엣지 3개)
말하며 진행: "고치기 전에 옆길도 한 번 볼게요. 이런 상황에선 어떻게 되나요."
- 빈칸으로 두면? 두 번 연달아 누르면? 새로고침하면? 세 시나리오만 같이 확인해 진짜 원인을 좁힌다.

### 5. 승인받고 한 곳만 수정
말하며 진행: "원인은 여기로 보여요. 여기만 이렇게 고칠게요, 진행할까요?"
- 어디를·왜·어떻게를 한 줄로 보이고 승인받는다. 승인 후 그 한 곳만 바꾼다. (plan.md 가 있으면 관련 항목만 갱신 / 대화 상대가 없는 자동 실행이면 이 한 줄을 기록으로 남기고 진행)

### 6. 같은 순서로 재검증
말하며 진행: "아까 1~2단계 그대로 다시 해서 증상이 사라졌는지 봅니다."
- 화면을 열어 직접 눌러 확인한다. e2e 테스트가 프로젝트에 있으면 같이 돌린다(없으면 넘어감).
- 실행 명령은 package.json 의 scripts 를 읽어 찾는다(하드코딩 금지).
- 안 사라지면 방금 수정을 되돌리고 원인 후보 2번으로 3단계부터 다시.

## 금지
- 원인 없이 "일단 이렇게 바꿔보자" 식 추측 패치.
- 한 번에 여러 파일·여러 곳 동시 수정.
- 에러 원문을 안 보고 넘겨짚기.
- .env 파일 열어 값 확인 (키 이름만 다룬다).
- 되던 것까지 건드려 "김에 정리".

## 3번 실패하면
같은 증상을 세 번 못 잡으면 멈춘다. 문제를 더 작게 쪼갠다: 화면을 반으로 나눠 어느 쪽에서 터지는지, 한 기능만 남기고 나머지를 잠깐 꺼서 범위를 좁힌다. 그래도 안 되면 사람에게(강사·커뮤니티) 증상 3줄과 지금까지 시도를 그대로 들고 물어본다.

## 졸업 한 줄
이 스킬 없이도 기억할 세 마디: 원인 먼저 → 한 곳만 → 다시 확인.
