# deploy: 배포하자

내 컴퓨터에서만 돌던 걸 인터넷 주소가 있는 진짜 사이트로 옮기는 일. 집에서 만든 요리를 가게에 내놓는 것과 같다. 재료를 빠뜨리면 손님상은 비어 나온다.

**철칙(Iron Law): 열쇠(.env)는 새 집에도 챙겨야 한다. 단, 값은 화면에 노출하지 않는다.** 로컬 `.env` 의 키를 하나도 빠짐없이 배포 서비스에 등록하되, 값은 절대 읽지도 출력하지도 않는다. 키 "이름"만 대조한다.

## 절차

### 1. 출발 점검
"올리기 전에, 지금 게 진짜 되는지부터 봅니다." 라고 말하고 진행한다.
- package.json 의 dev 스크립트를 찾아 로컬에서 화면이 뜨는지 확인한다.
- git 상태를 보고 최신 변경이 원격(push)에 올라갔는지 확인한다. 안 올라갔으면 "배포는 깃허브에 올라간 걸 가져가요. 먼저 올릴게요." 하고 push 한다.

### 2. 어느 폴더를 올릴지 정한다
"이 프로젝트는 폴더가 여러 개라, 배포 서비스한테 '이 폴더가 웹 앱이에요' 라고 짚어줘야 해요." 라고 말한다.
- 프로젝트 구조를 읽어 실제 웹 앱 폴더를 찾는다(여러 앱이 한 저장소에 있으면 Import 시 Root Directory 를 그 폴더로 지정하라고 안내).

### 3. 열쇠 챙기기 (.env 대조)
"집을 옮기니 열쇠도 다 챙겨요. 값은 제가 안 봅니다, 이름만 맞춰요." 라고 말한다.
- 로컬 .env 의 키 "이름" 목록만 뽑아, 배포 서비스에 전부 등록됐는지 대조한다.
- 브라우저(사용자 화면)에서 쓰이는 공개 키의 누락을 특히 짚는다 — 이 킷의 공개 키 명단은 `apps/web/next.config.mjs` 의 `env` 목록이다(`NEXT_PUBLIC_` 접두사는 이 킷에서 안 쓴다). 이게 빠지면 화면은 떠도 기능이 죽는다.

### 4. 열어서 눌러본다
"올라갔대도 진짜 되는 건 직접 눌러봐야 알아요." 라고 말한다.
- 배포된 실제 주소를 연다. 핵심 버튼 한두 개를 눌러 로컬과 같이 동작하는지 눈으로 확인한다.

### 5. 실패하면 넘긴다
빌드가 깨지면 "빨간 글씨(에러) 원문 그대로 가져와서 버그부터 잡을게요." 하고 에러 전문을 확보해 원인 수정으로 넘어간다.

## 금지
- .env 값을 읽거나 화면·로그에 출력하는 것 (키 이름만).
- 로컬에서 안 되는 걸 "일단 올려보자" 하는 것 (출발 점검 없이 배포 금지).
- 에러를 요약·의역해서 넘기는 것 (원문 그대로).
- 사용자 확인 없이 공개 범위·도메인 같은 되돌리기 어려운 설정을 건드리는 것.

## 막히면
빌드 에러가 길면 맨 위 첫 에러 한 줄만 떼어 그것부터 잡는다(한 번에 하나). 그래도 안 풀리면 에러 원문을 들고 강사·커뮤니티에 물어라.

## 졸업 한 줄
이 스킬 없이도 기억할 세 마디: 올렸나(push) → 열쇠 챙겼나(.env) → 열어서 눌러봤나.
