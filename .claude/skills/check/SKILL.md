# check: 되는지 확인

방금 만든 기능이 실제로 동작하는지 직접 눌러서 검증한다. 요리한 뒤 한 입 먹어보는 것과 같다. 안 먹어보고 손님상에 올리지 않는다.

## 철칙 (Iron Law)

**눈으로 "되겠지"는 확인이 아니다. 실제로 눌러본 것만 확인이다.** 코드를 읽어서 맞아 보인다는 건 근거가 아니다. 화면을 열어 그 동작을 손으로 재현해야 통과다.

## 절차

1. **확인할 동작 한 문장으로 적기.** 사용자에게 이렇게 말하며 시작하라: "먼저 뭘 확인할지 한 문장으로 정할게요. 예: '글 쓰고 저장 누르면 아래 목록에 바로 뜬다'. 이 한 문장이 합격 기준이에요." 방금 만든 것의 핵심 동작 1개를 이렇게 뽑는다.
2. **실행 방법 찾기.** package.json 의 scripts 를 읽어 dev·start 같은 실행 명령을 찾아 앱을 띄운다. "package.json 보고 실행 명령을 찾아서 앱을 켤게요"라고 말하며 진행하라. 명령을 지어내지 말 것.
3. **자동 손님(브라우저 로봇) 있는지 보기.** 프로젝트에 브라우저 자동 테스트가 있으면(playwright·cypress·e2e 폴더 등) 그 형식에 맞춰 1번의 시나리오를 한 개 추가한다. "이 프로젝트엔 자동 손님이 있으니 그 방식으로 시나리오를 넣을게요"라고 알린다.
4. **없으면 직접 눌러 확인.** 자동 테스트가 없으면 강요하지 말고, 화면을 열어 1번 문장 그대로 순서대로 눌러본다. "자동 테스트는 없으니 제가 직접 화면을 열어 눌러볼게요"라고 말하며 실제로 클릭·입력한다.
5. **초록/빨강 보고.** 됐으면 초록(무엇을 어떻게 눌러 확인했는지 한 줄), 안 됐으면 빨강(어디서 무엇이 기대와 달랐는지)으로 보고한다. "결과 알려드릴게요"라고 말한다.
6. **빨강이면 버그 잡기로.** 실패했으면 "버그 잡자 라고 말해 보세요"로 연결한다. 여기서 직접 고치려 뛰어들지 않는다.

## 금지

- 코드만 읽고 "되는 것 같아요"라고 통과 처리.
- 실행 명령을 package.json 확인 없이 지어내기.
- 자동 테스트가 없는데 억지로 새 테스트 프레임워크를 설치·강요.
- 여러 동작을 한꺼번에 확인하려다 뭐가 깨졌는지 못 짚기 (한 번에 동작 하나).
- .env 파일 열어보기 (키 이름만 다룰 것).

## 막히면

확인 범위를 더 작게 쪼개서(동작 하나를 더 잘게) 다시 눌러본다. 그래도 원인을 못 짚으면 강사·커뮤니티 같은 사람에게 화면과 증상을 그대로 들고 간다.

## 졸업

이 스킬 없이도 기억할 세 마디: **한 문장으로 정하고 → 실제로 눌러보고 → 초록이면 넘어가고 빨강이면 잡는다.**
