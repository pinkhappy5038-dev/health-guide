# 재료 & 오픈소스 활용법

밀키트의 핵심 = **직접 다 만들지 않는다. 남이 만든 걸 가져다 쓴다.** 이게 바이브코딩이 빠른 이유.

## 두 가지 "빌리기"

### 1. 서비스 빌리기 (SaaS · 키 필요)
남의 서버·기능을 빌린다. `.env` 에 키를 넣어 연결.

| 재료 | 무엇을 대신 | 언제 |
|------|-----------|------|
| **Supabase** | DB(창고) + 로그인(경비실) | 저장·회원가입 필요할 때 (제일 흔함) |
| **Google Sheets** | 시트를 DB처럼 | 아주 소규모·내부용, 진짜 DB 부담될 때 |
| **Cloudflare** | 배포·엣지·파일저장(R2) | 전세계 빠르게, 이미지 저장 |
| **Vercel** | 배포·호스팅 | 웹앱을 인터넷에 (깃허브 연결 자동) |
| **AI (Anthropic/OpenAI)** | 두뇌 (문구·이미지 생성) | 앱 안에서 AI 쓸 때 |

> 키에는 **두 종류**가 있다. **공개 키**(Supabase 의 `anon` 키 등)는 화면에 드러나도 정상이다. 원래 브라우저에서 쓰라고 만든 것이다. **비밀 키**(`service_role`·시크릿·결제 실키)는 클라이언트 코드에 절대 넣지 않는다. 결제 키는 **테스트 키부터** 쓴다. (자세한 보안 철학은 README 참고.)

### 2. 코드 부품 빌리기 (오픈소스 · 무료)
남이 만든 코드를 내 프로젝트로 가져온다.

| 방법 | 무엇 | 예 |
|------|------|-----|
| **npm 패키지** | 기능을 명령 한 줄로 설치 | `pnpm add date-fns` (날짜 처리) |
| **shadcn/ui** | UI 부품을 복사해 내 것으로 | `npx shadcn@latest add button` |
| **GitHub 레포** | 통째로 참고·복제 | 비슷한 프로젝트 찾아 구조 배우기 |

## 오픈소스 고르는 눈
- 직접 만들기 전에 **"이미 누가 만들었나"** 먼저 검색 (npm · GitHub).
- **별(star) 많고 최근 업데이트**된 것 우선 (죽은 프로젝트 피함).
- **라이선스** 확인 — MIT/Apache면 대체로 자유. (상업 사용 제한 있는 것 주의)
- 클로드에게 "이거 하는 라이브러리 추천해줘, 인기 있는 걸로" 물어도 된다.
